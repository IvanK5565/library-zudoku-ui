​Software Requirement Specification

​Introduction

​Appointment of a document

This document defines the functional and non-functional requirements for the "Electronic Diary" web application, developed using the Next.js and React frameworks. The goal is to create a tool for recording student learning activities, allowing teachers to assign grades and parents and students to view them.

​Scope

The system is intended for secondary general education institutions. Main users:

  • School administrators
  • Teachers
  • Students
  • Parents

​Definitions, abbreviations and acronyms

  • Next.js — framework for SSR/SSG React applications
  • SSR- Server-Side Rendering
  • Crud - Create, Read, Update, Delete
  • JWT - JSON Web Token for authorization
  • Rest Api - architectural style of client-server interfaces

​General Description

​Product prospects

The electronic diary is a separate web application with support for client-server architecture. The web client is implemented using React and Next.js, the server part is implemented via REST API (possibly with Node.js/Express or third-party API).

​Product functions

  • Registration and Authorization of users (JWT)
  • Teacher's panel: class creation, grades, statistics review
  • Student panel: Viewing grades, schedules, tasks
  • Parent panel: Viewing their children's grades
  • Administrator panel: User and class management

​User characteristics

  • Students - Basic PC Skills
  • Parents - basic skills, access to children's data
  • Teachers - average level, can create records
  • Administrators - high levels full of access

​Restrictions

  • Support for modern browsers (Chrome, Firefox, Safari, Edge)
  • SSR using Next.js
  • Using a database

​Assumption and addiction

  • Users have access to the Internet
  • The server part can be implemented by a separate team
  • Email verification or other security mechanisms are implemented separately

​Functional requirements

​Registration and Authorization

  • F1.1: The user can register (indicating the role)
  • F1.2: The user can be authorized
  • F1.3: jwt tokens to protect routes

​Teacher's panel

  • F2.1: Creating classes and subjects
  • F2.2: Adding grades
  • F2.3: Comments on grades
  • F2.4: Homework Loading

​The student panel

  • F3.1: View grades by dates and items
  • F3.2: Viewing homework
  • F3.3: Success Statistics

​Parents

  • F4.1: Binding to the student (by code)
  • F4.2: View Teachers' grades and comments

​Administrator panel

  • F5.1: ​​User management
  • F5.2: Role Presentation
  • F5.3: Class and Schedule Management

​Non-functional requirements

​Productivity

  • Pages should be downloaded in less than 2 seconds

​scalability

  • Support up to 1000 active users at the same time

​Safety

  • Authorization Protection through HTTPS and JWT
  • Validation of forms on the client and server

​The convenience of the interface

  • Intuitive design according to user role
  • Adaptive interface (mobile-first)

​Ability to extend

  • Ability to add modules (chat, notifications)

​Other requirements

  • use Typescript (recommended)
  • Apply component approach (React)
  • SSR/SSG for SEO and speed
  • CI/CD for automatic imprint (eg Vercel)
